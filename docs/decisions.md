即使某个决策后续被推翻，**不要删除历史记录**，而是新增一条说明变更。

## ADR-001: 选择 Astro 作为框架

**日期**：2026-04-20
**状态**：已采纳

**决策**：使用 Astro 作为静态站生成器。

**原因**：

- 教程网站的内容本质是"以文章为主 + 少量交互组件"，Astro 的 Islands 架构正是为此而生。
- 作者已有 Next.js 经验（AI 热点资讯平台项目），但 Next.js 是为"应用"设计，对纯内容站偏重。
- Astro 允许内容完全用 Markdown 写，作者写内容时完全不用碰代码。
- 编译产物是纯静态 HTML/CSS/JS，任何 Web 服务器都能托管，灵活性高。
- SEO 天然优秀，首屏性能极佳。

**替代方案**：

- **Next.js**：与已有项目同栈，上手快；但对内容站偏重。
- **纯静态 HTML/CSS/JS**：最简单，可作为教学案例；但维护多页面和共享组件会很快变成重复劳动。
- **VitePress / Docusaurus**：文档站专用；但风格难以脱离"技术文档感"，不适合本教程的生活化语气。

## ADR-002: 托管在自有腾讯云轻量应用服务器

**日期**：2026-04-20
**状态**：已采纳

**决策**：使用作者已有的闲置腾讯云轻量应用服务器，Nginx 原生部署，不用 Docker。

**原因**：

- 服务器已存在，边际成本为零。
- 对作者来说，亲手配 SSH/Nginx/systemd 比用 EdgeOne Pages 这类"一键托管"学到更多。
- 轻量服务器能跑任何东西，未来接评论、API 都不用迁移。
- 不用 Docker 是因为：Docker 本身又是一层黑盒，会掩盖"进程、端口、配置文件"这些基础概念——对作者的教学目标（**把黑盒拆开给小白看**）不利。

**替代方案**：

- **Vercel**：最方便，但国内访问不稳定，且掩盖了"服务器到底怎么运作"。
- **腾讯云 EdgeOne Pages**：国内版 Vercel，体验优秀，但同样隐藏底层。
- **腾讯云 COS 静态托管**：便宜，但需要手动上传或自己配 CI，麻烦程度比 SSH + git pull 更高。

## ADR-003: 不接域名，初期用公网 IP

**日期**：2026-04-20
**状态**：已采纳

**决策**：项目初期通过腾讯云公网 IP 访问（`http://[IP]`），不接自己的域名。

**原因**：

- ICP 备案周期 2-3 周，不阻塞内容开发。
- 公网 IP 本身是很好的教学素材：第 7 章讲"域名、DNS、HTTPS"时可以对着现状演示"它现在长什么样、加上域名后会长什么样"。
- 限制可以转化为叙事。

**未来计划**：Phase 6 接上备案域名后，Nginx 配域名 + Certbot 签 HTTPS 证书。

## ADR-004: 不使用 React，优先纯 Astro 组件

**日期**：2026-04-20
**状态**：已采纳

**决策**：交互组件优先用纯 Astro + 原生 JS 实现。确实需要状态管理时才引入 Svelte（不引入 React 和 Vue）。

**原因**：

- Astro 组件已经能覆盖 80% 的交互需求。
- Svelte 比 React 更轻，编译产物更小，更契合"性能优先"的内容站定位。
- 教学网站的组件通常不复杂，没有必要引入重量框架。

## ADR-005: 第 9 章的"30 分钟小工具"用简化版 AI 资讯平台

**日期**：2026-04-20
**状态**：已采纳

**决策**：第 9 章的"30 分钟小工具"实战项目是"AI 热点资讯平台的最小可演示版"——静态 JSON 假数据、硬编码的假 AI 摘要，只保留 Astro + Vercel（或腾讯云轻量）部署流程。

**原因**：

- 作者真实的 AI 资讯平台包含 6 个组件（Python 爬虫、Next.js、Supabase、Vercel、GitHub Actions、DashScope），小白 30 分钟不可能全部跑通。
- 砍到最小后，读者能真切体会"我写了个网站 + 它在互联网上能访问"的魔法时刻，成就感最重要。
- 第 10 章则告诉他们"现在做的东西，把假数据换成真爬虫、把硬编码换成真数据库、把假 AI 换成真 API，就是那个真实版本"——第 9 章给成就感，第 10 章给方向感。
