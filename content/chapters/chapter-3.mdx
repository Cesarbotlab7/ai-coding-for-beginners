## 你可能以为 GitHub 是这样的

也许你听说过 GitHub，大概知道"程序员都在上面发代码"。

有人以为它是程序员的朋友圈——发代码、涨粉丝、展示履历。

有人以为它是网盘——把代码存上去，跟百度云差不多。

这两个理解都不算错，但都没抓到最核心的一点：

**GitHub 是代码的时间机。**

---

## 你改坏了，怎么办

假设你正在开发 AI 资讯平台的前端，今天改了一个"按信源筛选"的功能。

改完觉得差不多了，关掉电脑去吃饭。

吃完回来，打开网页一看：首页空白了，什么都显示不出来。

你知道刚才改的代码破坏了什么，但你不记得原来是怎么写的了。

没有 GitHub？凉了。有 GitHub？三秒钟回到吃饭前的版本。

---

## 三个动作，理解 GitHub 的核心

**commit（提交）**：相当于在 Word 里点"保存"，但这次保存带了一张便签——你写下"为什么这次保存"。

> 示例：`feat: 加了按信源 Tab 筛选的功能`

**push（推送）**：把你本地的所有保存记录，同步到 GitHub 云端。就像把 U 盘里的文件上传到网盘，但 GitHub 存的是整个历史，不只是最新版。

**pull（拉取）**：把别人（或者你自己在另一台电脑）push 上去的更新，下载到本地。多人协作时，大家都往同一个仓库 push，再各自 pull 同步。

这三个词，程序员每天说几十次，但意思就这么简单。

---

## 为什么 commit 要写便签

一条好的 commit message，六个月后救你自己的命。

当你三个月后翻历史，看到一条 `fix: 修了个东西`，你会完全不知道它改了什么。

但如果你当初写的是 `fix: 修复 Supabase 查询时 url_hash 重复导致的 409 报错`，你立刻就能锁定那次改动。

commit message 是写给未来的自己和队友的。

## GitHub 能干的不只是存代码

GitHub 还有几个你会经常用到的功能：

**Issues（议题）**：记 bug、记需求、记待办。就像 Notion 的任务卡，但直接挂在代码仓库旁边。

**GitHub Actions（自动化）**：定义一套"当代码 push 之后，自动做什么"。AI 资讯平台的爬虫就靠它每 3 小时自动跑一次——不需要你一直开着电脑守着。

**GitHub Pages**：把一个静态网站直接部署在 GitHub 上，免费，适合文档或展示页。
