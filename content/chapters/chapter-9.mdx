## 上线了，然后呢

你做好了，推上去了，发朋友圈了。

然后呢？

99% 的个人项目死在"然后"这一步——因为没有人告诉你，上线之后该干嘛。

真实答案是：**日常维护**。

开了一家店，你要每天看营业额、修坏掉的设备、换季换菜单。软件项目也是一样——只是"营业额"叫做访问量，"坏掉的设备"叫做 bug，"换菜单"叫做迭代新功能。

## 三件你会反复做的事

**看数据**

你看的不是 KPI，而是信号：有没有人来、来了有没有报错、响应有没有变慢。

免费工具：Vercel Dashboard 自带访问统计、GitHub Actions 日志、Sentry（免费额度足够个人项目）。

**修 bug**

bug 的来源有三条路：
1. 用户告诉你（朋友抱怨、自己用的时候发现）
2. 日志报错（服务器或者 CI 的 log 里有 error）
3. 监控告警（某个错误率突然升高）

定位 bug 的标准流程：读报错 → 找到那一行代码 → 理解它为什么在这种情况下出错 → 改 → 测 → 推上去。

**迭代**

迭代的本质是：**你做完之后，通过真实使用或者用户反馈，发现了更好的做法**。

AI 资讯平台上线后有过三次典型迭代：

- **第一次**：爬虫挂了，是靠 GitHub Actions 的邮件通知发现的——Actions workflow 失败会发邮件给仓库 owner。当时的问题是某个 RSS 源的 XML 格式变了，`feedparser` 解析出空结果，但代码没有报错，只是悄悄写入了空数据。
- **第二次**：想加 arXiv 作为新信源。arXiv 有官方 API，直接调就行——但返回的论文标题全是英文，AI 摘要配额跑超了。解决方案是只对 tier1、tier2 的推文内容生成摘要，arXiv 只存原标题。
- **第三次**：发给朋友用，对方说"手机上标题太长了，看不完"。花了二十分钟加了一行 CSS `text-overflow: ellipsis`——这是前端的事，和后端无关。

## 这个循环，会一直持续

没有"做完了"这件事。

每一次上线，都是开始收集下一轮反馈。每一次迭代，都是在上一个版本的基础上往前走一步。

好消息是：**随着时间推移，每次迭代的质量会越来越高、速度会越来越快**——因为你对自己的产品越来越了解了。
