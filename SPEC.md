# SPEC.md — WorkBuddy 保姆级入门教程

本文档是网站产品、内容和交互的唯一功能规格。实现与本文档冲突时，以本文档为准。

## 一、产品定义

### 一句话定位

一个面向完全零基础用户的中文 WorkBuddy 交互教程：从没安装过，到能安全地完成第一个文件任务，再到把已验证的流程交给手机助理和自动化。

### 用户完成教程后能做到

1. 从官方入口下载安装 WorkBuddy，并核对系统和版本。
2. 为任务创建独立工作区，理解备份、路径和权限边界。
3. 分清默认、Plan、Ask 三种任务模式和默认、完全访问两种权限。
4. 把模糊要求写成包含目标、素材、步骤、输出和验收标准的任务。
5. 找到产物并人工核对数量、格式、内容和保存位置。
6. 管理记忆，分清 Skill、专家和专家团。
7. 先手动跑通，再配置远程助理和自动化。

### 不在范围内

- 不教授 AI 编程、前端、后端、数据库、服务器、DNS 或 GitHub。
- 不替用户真实操作 WorkBuddy、修改本地文件或绑定第三方账号。
- 不把合同审查、网页抓取、批量删除等高风险任务作为新手主线。
- 不承诺固定积分、技能数量、模板数量或永久不变的界面入口。
- 不复制外部教程全文或整套截图；外部材料只用于事实核验和教学结构参考。

## 二、信息来源与版本策略

事实优先级：

1. WorkBuddy 当前客户端真实验收。
2. WorkBuddy 官方文档和更新日志。
3. 程序员小灰的两篇 X 教程，用于教学顺序、案例和踩坑参考。

每章 frontmatter 必须包含：

- `verified`: 最近核验日期。
- `verified_version`: 最近核验的 WorkBuddy 版本范围。
- `sources`: 本章主要事实来源链接，仅作为内部核验元数据，不在公开页面展示。

容易变化的积分、套餐、模型和市场规模不写死；需要时链接官方页面。界面名称发生变化但流程仍成立时，先更新标注和替代路径；流程失效时，本章显示版本提醒。

## 三、网站结构

### 路由

| 路径 | 用途 |
|---|---|
| `/` | 首页、课程地图、本地学习进度 |
| `/chapter/[slug]` | 11 个课程单元 |

网站保持纯静态输出，不要求登录，不把学习进度上传服务器。

### 首页

首页按以下顺序展示：

1. Hero：产品名、适用人群、完成结果。
2. “你会亲手完成什么”：第一次文件任务和自动化毕业任务。
3. 课程进度：完成单元数、继续学习入口、重置入口。
4. 5 个阶段和 11 个课程卡片。
5. 作者和版权信息。

### 章节页

章节页包含：

1. 返回目录、上一章、下一章。
2. 阶段、单元编号、标题、学习目标、阅读时间、核验版本和日期。
3. 正文与一个核心交互实验。
4. “现在应该看到什么”和失败分支。
5. 本章要点。
6. 标记完成按钮。
7. 下一章入口。

## 五、课程交互规格

### 0. 建议与交付对比模拟器

- 同一个任务可切换“普通聊天”和“WorkBuddy 任务”。
- 普通聊天只显示建议。
- WorkBuddy 任务显示理解目标、拟定步骤、请求权限、生成产物、人工验收五个环节。
- 用户必须能用按钮和键盘完成切换。

### 1. 双系统安装向导

- 首先选择 Windows 或 macOS。
- 每个平台展示环境要求、官方下载、安装、扫码登录、检查更新五步。
- 不模拟真实下载，不自动打开安装包。
- 每一步展示成功信号和常见卡点。
- 安装入口必须指向官方页面。

### 2. 工作区安全实验

- 用户在三个候选位置中选择工作区。
- 系统目录、包含原始重要资料的目录给出风险提示。
- 独立练习目录给出推荐说明。
- 用户需完成“只放副本”“保留原件”“知道产物在哪”三项检查。

### 3. 模式和权限决策实验

- 工作模式和权限模式必须分成两组控件，禁止混写。
- 按当前客户端路径展示：“更多操作 → 模式”包含默认模式、Plan、Ask；输入框下方单独显示权限。
- 至少提供四种任务：纯问答、多步骤文件任务、简单可恢复任务、高风险任务。
- 规则由 `src/lib/workbuddy-labs.ts` 提供。
- 系统永不自动推荐完全访问权限。

### 4. 界面热点图

- 用自制 HTML/CSS 高保真重建当前 WorkBuddy 客户端，不复制真实账号、任务和文件内容。
- 统一外壳必须覆盖：左侧导航与任务历史、中间新建/执行区、右侧产物区、工作空间、模式、权限、模型、技能。
- 每个热点使用真实按钮，支持 Tab、Enter 和 Space。
- 不依赖第三方截图才能理解。
- 手机端保持桌面客户端比例，仅在外壳内部左右滑动；页面本身不得横向溢出。

### 5. 完整任务模拟器

主线任务：把虚构的 `meeting-notes.txt` 整理成 `meeting-summary.md`。

顺序固定：

1. 选择独立练习工作区。
2. 添加样例会议记录。
3. 从“更多操作 → 模式”开启 Plan，并保持默认权限。
4. 查看并确认四步计划。
5. 模拟执行。
6. 打开产物。
7. 进入验收。

用户未确认计划前，不得直接显示完成状态。

### 6. 五要素提示词生成器

字段：

- 目标
- 输入素材
- 处理要求
- 输出格式
- 验收标准

每项 20 分。空白和纯空格均视为未填写。生成内容可复制，但复制是用户主动操作，不自动写入系统剪贴板。

### 7. 产物验收实验

给出一份故意有缺陷的模拟产物，用户检查：

- 文件名和格式
- 必填栏目
- 事实是否来自原始素材
- 负责人和截止时间
- 保存位置

只有找出全部问题，才显示“验收通过”。发现问题后给出修正指令草稿。

### 8. 记忆审阅器

每条模拟记忆可选择：

- 保留
- 改写
- 删除

至少包含过时偏好、敏感信息、稳定偏好和未经核验事实四类。组件强调：

- 记忆可能概括错误。
- 重要判断前应回看原始信息。
- 不需要记住的隐私应删除或关闭记忆。

不展示旧版内部文件路径，不用虚构 Token 占用条。

### 9. 能力配对游戏

用户把场景匹配到：

- Skill：稳定、可复用的操作方法。
- 专家：领域判断和输出标准。
- 专家团：多个角色按流程协作。

答错后解释原因；完成后演示在任务输入框选择或调用 Skill。技能数量等市场数据不写死。

### 10. 上线检查与故障实验

毕业任务：每日 AI 资讯早报。

顺序固定：

1. 手动运行一次。
2. 人工检查来源、时间范围、条数和输出格式。
3. 使用绝对路径。
4. 确认电脑不会休眠且 WorkBuddy 保持运行。
5. 配置手机助理或小程序推送。
6. 开启定时任务。
7. 第二天检查执行历史。

未满足条件时一次返回全部问题。组件只模拟配置，不绑定真实账号或创建真实任务。

## 六、全局交互规则

### 学习进度

- 使用 `localStorage`，键名为 `workbuddy-course-progress-v1`。
- 数据只保存已完成 slug 数组。
- 首页显示完成数量和继续学习入口。
- 用户可主动重置，重置前二次确认。
- 无 JavaScript 时课程正文和导航仍可使用。

### 可访问性

- 所有交互可仅用键盘完成。
- 状态变化使用 `aria-live`。
- Tab、radio、checkbox 等采用正确原生控件或完整 ARIA 模式。
- 焦点清晰可见。
- 支持 `prefers-reduced-motion`。
- 颜色不是唯一的状态信号。

### 安全

- 不要求用户粘贴 API Key、密码、Cookie 或真实合同。
- 所有练习数据均为虚构数据。
- 涉及多步骤或边界不清的文件修改时默认使用副本、Plan 和默认权限。
- 完全访问权限只解释风险，不作为教程推荐路径。

## 七、验收标准

### 自动化

- 单元测试覆盖课程顺序、提示词评分、模式推荐和自动化上线条件。
- E2E 覆盖首页进度、模式实验、提示词生成和完整第一单。
- `npm run test:unit` 通过。
- `npm run build` 通过。
- `npm run test:e2e` 通过。

### 页面

- 360px 宽度页面无横向滚动；WorkBuddy 外壳允许内部横向滑动。
- 浅色、深色和 reduced-motion 可用。
- 无重复 DOM id、控制台错误和内部断链。
- 每章恰好包含一个核心交互。
- Lighthouse 性能、可访问性、SEO 不低于 90。

### 内容

- 每一步都有操作、成功信号和失败分支。
- 易变事实带官方链接或核验日期。
- 不把作者经验写成官方硬限制。
- 不宣称 AI 能替代律师、财务或其他专业责任人。
